Redis中的配置文件详解

发表于2022-08-01|更新于2022-08-02|后端

Redis的配置文件所在位置：/etc/redis/redis.config

Redis支持的数据度量单位

Redis的配置文件开头部分：

该部分表示Redis是只支持bytes为单位的，即以字节为单位

最后一行表示数据单元的书写是对大小写不敏感的，1GB、1Gb和1gB都可以用来表示

继续往下走，来到INCLUDES部分

配置文件之 INCLUDES

在多实例的情况下，Redis可以包含其他公用的配置文件

Redis本身只有一个配置文件，但是在需要进行多人开发维护的时候，就需要多个配置文件

此时就可以通过include /path/to/local.conf这样的方式配置进来，原本的redis.conf作为总配置文件用于管理

并且不能写在redi.conf的开头，会覆盖引入文件的配置，所以想以引入文件的配置为主

就需要将include配置写在redisconf文件的末尾

下面的MODULES先暂时跳过

来到比较重要的NETWORK的配置

配置文件之 NETWORK

bind

初始的配置中bind被配置为bind: 127.0.0.1::1，表示只能在本机访问，如果安装在虚拟机上，那么只能在虚拟机上进入Redis

我这里是没有注释的，所以只能本机访问，因为虚拟机的命令行交互不是太方便，所以使用Xshell进行配置连接

为了能够在主机上连上Redis，可以将其注释掉，这样所有的机器都能通过本机的IP地址以及端口6379进行连接（这样是不安全的）

当然在生产环境下，应该写应用服务器的地址，使得只有服务器可以远程访问

protected-mode

其次继续往下可以找到protected-mode，译为保护模式，默认为yes

如果开启了该配置，在没有设定bind且没有设置Redis密码的情况下，Redis只允许接收本机（指安装Redis的机器）的连接

如果需要使远程服务器可以访问，则需要将protected-mode: yes改为protected-mode: no

port

继续往下，是Redis占用的端口号，默认为6379，一般情况不需要更改

tcp-backlog

再往下就是tcp-backlog配置，用来设置tcp的backlog，默认为511

backlog是一个连接队列，backlog队列的总和 = 未完成三次握手队列 + 已经完成三次握手队列

在高并发环境下，backlog的值需要很大，来满足客户端的大量连接需求，避免客户端连接缓慢

Linux内核会将该值减小到/proc/sys/net/core/somaxconn的值（128）

所以如果需要增大backlog的值，就增大/proc/sys/net/core/somaxconn和/proc/sys/net/ipv4/tcp_max_syn_backlog两个值

timeout

继续下面的是timeout，表示连接Redis的超时时间，默认为0

比如一个场景中，某客户端连上Redis，但是长时间没有操作它，如果该时间大于配置的timeout

就会自动断开连接，无法继续操作，需要重新连接，如果该值配置为0，表示永不超时，以秒为单位

tcp-keepalive

最后是tcp-keepalive，检测客户端是否还在连接，值默认为300

在TCP协议中也有该属性，在Web1.0时代，以前的网站发送请求给服务端，服务端响应后，连接就会中断

但是一个网站除了有文本内容外，还有js、css以及图片等静态文件，只有所有资源都加载完成，才能看到网站的全貌

但是一个网站可能会有几十个js、css等请求，每请求一次就建立连接，消耗的资源过大

所以需要连接需要在短时间内得到复用，在加载同一个网页时，复用连接，这就使用到了keepalive属性

而在Redis中同样如此，客户端向服务端请求需求时，可能会从Redis数据库中请求多个资源，那么如果多次连接请求

代价会非常大，而keepalive就会每过N秒检测一次，客户端是否还有其他请求，如果没有则关闭连接

如果将值设置为0，则不会进行keepalive检测

配置文件之 GENERAL

daemonize

表示Redis是否为守护进程，能否在后台启动运行，默认为no

需要改为yes，否则不方便进行学习操作

pidfile

在开启Redis的守护进程后，将Redis的进程pid保存到指定的文件中，默认为/var/run/redis/redis-server.pid

loglevel

表示指定Redis的日志级别，默认为notice

Redis总共支持四个级别：debug，verbose，notice，warning

  • debug：Redis会给出详细的操作信息，对于开发很有用

  • verbose：Redis给出一些有用的信息

  • notice：是中等程度的verbose，可能是生产环境下有用的

  • warning：只会给出有用并且重要的信息

生产环境建议选择notice或warning

logfile

设置Redis日志文件的输出位置，默认为空

我这里是设置为：

databse

设置Redis的数据库数量，默认为16，标号从0~15

always-show-logo

启动Redis服务的时候，显示一个用ASCII码表示的一个艺术logo，多新鲜，默认为yes

配置文件之 SECURITY

requirepass

表示设置Redis的密码，该配置默认是被注释的，表示没有密码

如果需要设置密码（建议设置），可以将该注释取消，并且修改其后面的值

还可以通过config set requirepass "密码"来设置（没有Redis密码的情况下连接Redis，然后进行该操作）

通过config get requirepass来得到密码（这是连接上Redis才有的操作）

Redis在设置密码后，需要使用auth 密码来进行登陆验证，如果Redis返回OK，表示连接成功

配置文件之 CLIENTS

maxclients

表示Redis可以同时连接客户端的最大数量，默认是被注释的，默认为10000

如果达到的限制数量，会拒绝客户端的请求连接

配置文件之 MEMORY MANAGEMENT

这里就不附带源码，因为全是注释，可以自行查看内容

maxmemory

表示Redis可以使用的内存量，默认是被注释的

如果达到内存使用上限，Redis会试图移除内部数据，移除规则可以通过配置项maxmemory-policy来指定

maxmemory-policy

表示Redis中的移除策略，默认被注销

  • volatile-lru：使用LRU算法（最近最少使用）移除key，只对设置了过期时间的键进行移除

  • allkeys-lru：所有集合key中，使用LRU算法移除key

  • volatile-random：设置过期时间的集合key中，随机移除key

  • allkeys-random：所有集合key中，随机移除

  • volatile-ttl：移除TTL值（Time To Live 生命周期）最小的key，即最近要过期的key

  • noeviction：不进行移除，针对写操作，只返回错误信息

maxmemory-samples

设置Redis的内存样本数量，LRU算法和最小TTL算法并非精确的算法，而是估算值

所以可以设置样本的大小，Redis会默认检查这么多的key并选择其中LRU的那个

一般是3到7的数字，数值越小样本越不准确，但是性能消耗越小
